# YouTube 脚本生成器 (@einverne 技术频道)

专为 @einverne 技术频道设计的视频脚本生成工具，将技术主题或 Obsidian 笔记转化为专业、详细的视频拍摄脚本。

## 核心功能

生成包含以下元素的完整视频脚本：
- 时间轴标记（精确到段落）
- 逐字稿和讲解要点
- 镜头提示（主镜头、屏幕录制、特写等）
- B-roll 素材建议
- 字幕要点标注
- 互动元素设计
- 制作清单

## 频道规格

**固定参数：**
- 频道：@einverne
- 主持人：EV
- 目标时长：8-15 分钟
- 目标受众：初学者和中级开发者
- 主要语言：中文
- 固定开场白："大家好，我是 EV，欢迎来到我的频道。"
- 固定结束语："欢迎大家点赞、关注、转发。"

**风格定位：**
- 前半段：技术分享（轻松、对话式）
- 后半段：正式教学（结构化、步骤清晰）

**默认保存路径：**
- 脚本文件默认保存到：`~/Sync/wiki/EVLabs/`
- 文件命名格式：`脚本-[主题简称]-[日期YYYYMMDD].md`
- 示例：`脚本-Docker-Compose-教程-20250120.md`

## 脚本生成工作流

### 第 1 步：分析输入内容

**接受的输入格式：**
1. **主题词**：简短的技术主题（如"Docker Compose 教程"）
2. **Obsidian 笔记**：包含 Markdown 格式和双链的技术笔记
3. **要点列表**：关键知识点的列表
4. **技术文档**：已有的技术说明文档

**分析任务：**
- 识别核心技术/概念
- 确定适用场景和目标受众
- 评估内容复杂度
- 判断是否需要实操演示
- 提取关键知识点

**如果输入内容不足以生成完整脚本：**
- 向用户提出具体问题：
  - 需要讲解的深度（入门/进阶）
  - 是否需要代码演示
  - 重点关注的方面
  - 有无特定的使用场景

### 第 2 步：确定脚本结构

根据内容类型选择合适的结构模式：

**教程类（有实操步骤）：**
```
开场段 (1分钟) → 概念介绍 (2.5分钟) → 准备工作 (1.5分钟)
→ 实战教程 (7分钟) → 验证测试 (1.5分钟) → 扩展延伸 (1分钟) → 结尾 (0.5分钟)
```

**概念讲解类（偏理论）：**
```
开场段 (1分钟) → 概念介绍 (3分钟) → 工作原理 (4分钟)
→ 使用场景 (3分钟) → 最佳实践 (2.5分钟) → 扩展延伸 (1分钟) → 结尾 (0.5分钟)
```

**工具介绍类：**
```
开场段 (1分钟) → 工具介绍 (2分钟) → 核心功能 (3分钟)
→ 实战演示 (5分钟) → 进阶技巧 (2.5分钟) → 对比总结 (1分钟) → 结尾 (0.5分钟)
```

**对比分析类：**
```
开场段 (1分钟) → 背景介绍 (2分钟) → 选项A分析 (3分钟)
→ 选项B分析 (3分钟) → 对比总结 (2.5分钟) → 选择建议 (2分钟) → 结尾 (0.5分钟)
```

**参考资源：**
查看 [references/structure-guide.md](references/structure-guide.md) 了解详细的段落结构标准和时长分配策略。

### 第 3 步：内容填充与扩展

**对于每个段落，生成：**

**1. 主要内容（讲解词）：**
- 采用口语化表达
- 包含过渡语和连接词
- 标注语速变化和停顿
- 体现风格切换（分享模式 vs 教学模式）

**2. 镜头提示：**
- 主镜头：人像拍摄的构图和表情提示
- 屏幕录制：需要录制的具体操作
- 特写：需要放大显示的元素
- 画中画：是否需要人像叠加在屏幕录制上
- 标注：需要圈选、高亮、箭头指示的位置

**3. B-roll 素材建议：**
- 列出需要的补充画面
- 说明素材用途
- 建议素材来源

**4. 字幕要点：**
- 关键技术术语（中英文对照）
- 重要命令/代码
- 核心概念定义

**5. 互动元素：**
- 提问点
- 评论引导
- 参与召唤

**参考资源：**
- 技术内容呈现：查看 [references/technical-presentation.md](references/technical-presentation.md)
- 观众互动技巧：查看 [references/engagement-techniques.md](references/engagement-techniques.md)

### 第 4 步：质量优化

**检查清单：**

**内容完整性：**
- [ ] 开场是否包含效果预览和价值主张
- [ ] 概念解释是否清晰（是什么、为什么、怎么用）
- [ ] 教程步骤是否有明确的编号和目标
- [ ] 每个步骤是否包含预期结果和常见错误
- [ ] 是否提供了验证测试流程
- [ ] 是否有扩展学习建议
- [ ] 结尾是否包含固定元素

**难度适配：**
- [ ] 术语是否都有解释
- [ ] 步骤是否足够详细（初学者能跟上）
- [ ] 是否标注了可跳过的进阶部分（中级开发者不觉得慢）

**互动设计：**
- [ ] 前 30 秒是否有吸引力
- [ ] 是否有 2-3 个互动引导点
- [ ] 是否使用"我们"而非"你"建立连接感
- [ ] 是否有适当的提问

**制作可行性：**
- [ ] 镜头提示是否清晰可执行
- [ ] B-roll 需求是否现实（可获取）
- [ ] 时长分配是否合理（8-15 分钟范围内）

**语言表达：**
- [ ] 是否口语化、自然
- [ ] 是否避免了教条式说教
- [ ] 过渡是否流畅
- [ ] 关键点是否有强调标注

### 第 5 步：输出脚本

**输出格式：**

使用 [assets/script-template.md](assets/script-template.md) 作为基础模板，填充所有内容。

**脚本应包含：**

1. **头部信息**
   - 视频标题
   - 主题简述
   - 目标时长
   - 目标受众

2. **分段脚本**（按时间轴组织）
   - 每段标题带时间轴标记 `[00:00-01:00]`
   - 完整的讲解词
   - 详细的镜头提示
   - 具体的 B-roll 素材清单
   - 字幕要点列表

3. **制作清单**
   - 拍摄前准备
   - 拍摄中检查
   - 后期制作要点
   - 发布准备

4. **附加说明**
   - 预计制作时间
   - 素材来源
   - 特殊注意事项

**保存脚本：**
生成完整脚本后，立即保存到指定目录：
- 使用 Write 工具保存文件
- 路径：`~/Sync/wiki/EVLabs/脚本-[主题简称]-[日期].md`
- 日期格式：YYYYMMDD（例如：20250120）
- 保存成功后，告知用户完整的文件路径

## 使用示例

### 示例 1：基于主题词生成

**输入：**
```
主题：Docker Compose 快速搭建开发环境
```

**处理流程：**
1. 分析主题：这是一个实操教程类内容
2. 选择结构：开场 → 概念介绍 → 准备工作 → 实战教程 → 验证测试 → 扩展延伸 → 结尾
3. 内容扩展：
   - 概念：Docker Compose 是什么，为什么用它
   - 准备：需要的工具和版本
   - 实战：分步骤演示配置过程
   - 验证：测试运行结果
   - 扩展：性能优化和最佳实践
4. 填充细节：每步添加镜头提示、B-roll、字幕要点
5. 输出完整脚本

### 示例 2：基于 Obsidian 笔记生成

**输入：**
```markdown
# [[Kubernetes]] 基础概念

## 核心组件
- [[Pod]]：最小部署单元
- [[Service]]：服务发现和负载均衡
- [[Deployment]]：应用部署管理

## 架构
Master 节点管理整个集群...
[[Worker Node]] 运行实际的应用容器...

## 使用场景
适合需要容器编排的场景...
```

**处理流程：**
1. 提取关键信息：
   - 主题：Kubernetes 基础
   - 双链关键词：Pod, Service, Deployment, Worker Node
   - 结构：概念讲解 + 架构说明 + 场景分析
2. 判断内容类型：概念讲解类
3. 选择结构：开场 → 概念介绍 → 工作原理 → 使用场景 → 最佳实践 → 扩展 → 结尾
4. 扩展笔记内容：
   - 用类比解释抽象概念
   - 添加图示说明
   - 补充实际例子
5. 输出脚本

## 特殊场景处理

### 场景 1：内容过长（超过 15 分钟）

**压缩策略：**
1. 简化概念介绍，聚焦核心
2. 减少代码逐行解释，只讲关键部分
3. 合并相似步骤
4. 将扩展内容提炼为要点，详细内容引导到评论区或下期视频

**调整示例：**
```
原：每个配置项逐一解释（5分钟）
改：分组解释配置项，只详细讲最重要的 2-3 个（2.5分钟）
```

### 场景 2：内容过短（少于 8 分钟）

**扩展策略：**
1. 增加实际应用场景和案例
2. 添加常见问题和解决方案
3. 补充最佳实践和优化技巧
4. 增加与替代方案的对比
5. 添加进阶使用技巧

### 场景 3：高度技术性内容（难度较大）

**降低门槛策略：**
1. 使用生活化类比解释抽象概念（参考 technical-presentation.md 中的类比库）
2. 增加视觉辅助（图表、动画、流程图）
3. 分层讲解：先讲表面用法，再讲工作原理
4. 提供简化版和完整版两种实现
5. 在关键点停顿，给观众理解时间

### 场景 4：多个子主题

**组织策略：**
1. 在开场明确告知视频包含的所有子主题
2. 为每个子主题分配独立的段落
3. 在段落之间添加清晰的过渡
4. 在描述中添加时间戳，方便观众跳转
5. 结尾时整体回顾各子主题的关联

## 注意事项

### 必须遵循
- ✅ 使用固定的开场白和结束语
- ✅ 前半段保持轻松对话感，后半段转向结构化教学
- ✅ 所有技术术语首次出现时必须解释
- ✅ 为初学者提供足够的细节，为中级开发者标注可跳过部分
- ✅ 口语化表达，避免书面语
- ✅ 使用"我们"而非"你"建立连接感

### 避免
- ❌ 跳过基础步骤
- ❌ 使用未解释的行话
- ❌ 说"这很简单"（可能对新手不简单）
- ❌ 连续 5 分钟相同类型内容（保持节奏变化）
- ❌ 突然结束没有收尾
- ❌ 生硬的"请点赞关注转发"（自然融入）

### 优化建议
- 💡 在前 30 秒展示效果或价值
- 💡 每 2-3 分钟改变一次画面/节奏
- 💡 至少设置 2-3 个互动点（提问、征求意见）
- 💡 提供进度提示（"我们现在完成了三分之一"）
- 💡 在难点前做好预期管理（"这一步稍微复杂，不过别担心"）
- 💡 遇到错误不要剪掉，演示如何调试
