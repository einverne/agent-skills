# Wiki 文章编译指南

`wiki/<领域>/` 下文章的写作规范。这些文章是知识库的核心产出，也是 query 操作的回答依据。

## 目标特征

- **长度：** 2000-4000 字。超过时拆分为子文章。
- **受众：** 对该领域有一定基础的读者。首次出现的专业术语需解释。
- **独立性：** 读者仅凭一篇文章就能理解该主题。
- **密集双链：** 每篇文章目标 10-30 个 `[[wikilink]]`。
- **有据可查：** 每个事实性论断都追溯到 frontmatter `sources:` 中列出的原始文件。
- **实体类型匹配：** 按 `entity_type` 选择章节结构（见 `references/entity-types.md`）。

## 语气与风格

- **领域知识，非个人笔记。** 写成任何人都能参考的资料。不要出现第一人称。
- **陈述性、技术性、中立。** 避免 hype，避免模糊措辞。直接陈述事实，附上来源。
- **具体优先。** 能用代码块、表格、图表说清的，不要用大段散文。

## 章节结构

### 按实体类型选择章节

不同实体类型有不同的必需章节。详见 `references/entity-types.md`。

**所有类型通用：**
1. **H1 标题** — 与文件名一致
2. **导语段** — 2-4 句话，说明主题是什么、为什么重要、本文覆盖范围
3. **核心章节**（H2）— 按实体类型的必需章节组织
4. **来源与延伸阅读** — 列出所有引用来源的 wikilink

### 可选章节（所有类型通用）

- **对比表格** — 调研多种方案时
- **代码示例** — 可运行的代码片段
- **架构图** — ASCII 或 Mermaid
- **权衡分析** — 设计上有张力时
- **未来方向** — 领域发展趋势
- **Counter-Arguments & Data Gaps** — 反论与数据缺口（见下文）

## Counter-Arguments & Data Gaps

> 来源：社区建议 — 每次 LLM 更新概念页面时，生成一个隐式检验段落，防止偏见积累。

**何时必需：**
- 综合了 3+ 来源的文章 → 建议添加
- 主题具有争议性或多种观点 → 必需
- Lint 时检查：若缺少且符合条件，标记为待改进

**格式：**
```markdown
## Counter-Arguments & Data Gaps

### 对立观点
- 针对本文核心论点的最强反驳
- 引用来源（如有）

### 数据缺口
- 本文论断缺少直接证据的部分
- 需要补充来源的方向

### 可信度评估
- 本文 confidence: high/medium/low
- 依据：来源数量、来源权威性、论据一致性
```

## 来源引用与验证

> The human owns verification. — bluewater8008

- **每篇文章必须引用真实来源。** 不要仅凭通用知识编写。如果语料库中没有，要么 ingest 新来源，要么不写该论断。
- **Frontmatter `sources:`** 列出所有参考的原始文件，以 wikilink 形式。
- **正文内引用** 允许但非必需。底部的 Sources 章节是必需的。
- **直接引用** 需要加引号并标注来源。
- **编译后验证：** LLM 可以综合而不引用来源。每次编译后建议用户抽查关键论断。

## 双链密度

双链是知识图谱的边。每个相关概念的首次提及都应该是 wikilink：

- 提及另一个概念文章 → `[[概念名]]`
- 提及有独立文章的工具/框架 → `[[工具名]]`
- 跨领域引用 → `[[领域/文章名]]`

不要对常见词过度双链。不要对没有独立文章的人名/组织名双链。

## Token 预算感知

编译时注意上下文消耗。遵循渐进加载原则：

| 级别 | Token 预算 | 何时加载 |
|------|-----------|---------|
| L0 | ~200 | 每次会话：项目上下文（SKILL.md frontmatter） |
| L1 | ~1-2K | 会话开始：Concept Index + Dashboard |
| L2 | ~2-5K | 编译/查询时：搜索结果摘要 |
| L3 | 5-20K | 深度操作：完整阅读原始资料或 Wiki 文章 |

先读索引定位，再读文章全文。不要在未查索引时直接读全部文章。

## 反模式

- 文章仅总结单一来源 → 应综合多个来源
- 没有入链的孤儿文章 → 每篇文章都应通过链接图可达
- 没有出链的文章 → 每篇文章都应参与图谱
- 泛泛而谈、放之四海皆准的内容 → 要具体到该主题的术语、模式、张力
- 大段重述前置知识 → 链接到前置文章即可
- 所有文章结构相同 → 按 entity_type 选择不同章节
- 无条件信任 LLM 的综合 → 抽查来源引用的准确性

## 更新已有文章

1. 加载当前文章完整内容
2. 加载上次编译后新增的原始资料
3. 识别变化（新技术、修正、新术语）
4. **用结构化 diff 展示每处修改，逐条确认：**

   > **当前：** `原文内容`
   >
   > **拟改为：** `新内容`
   >
   > **原因：** `为什么改`
   >
   > **来源：** `[[raw/xxx.md]]`

5. **矛盾扫描：** 如果新信息与 wiki 中已有内容矛盾，grep 所有文章查找矛盾点并全部修正
6. **下游影响检查：** grep 查找所有链接到被更新文章的页面，确认是否需要同步更新
7. 原地更新文章，尽可能保持结构
8. 更新 frontmatter 中的 `updated:` 日期
9. 添加新的 `sources:` 条目
10. 检查现有双链是否仍然有效，为新引入的概念添加双链

## 文章拆分条件

- 字数超过 4000
- 单个 H2 章节超过 1000 字
- 文章覆盖两个不同子主题，各自值得独立条目
- 多个其他文章需要链接到某个子章节

拆分后，父文章 wikilink 到子文章，并更新索引。

## 写新文章 vs 扩展已有文章

**写新文章：**
- 3+ 篇已有文章双链到一个不存在的概念
- query 反复综合相同的跨文章内容
- 该主题有独立来源、模式和术语

**扩展现有文章：**
- 新材料是对已有概念的细化、示例或子方面
- 子主题独立成文不足 500 字
